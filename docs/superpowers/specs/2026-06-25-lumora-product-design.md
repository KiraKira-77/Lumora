# 光枢 Lumora 产品方案设计 V1

## 1. 产品结论

光枢 Lumora 的 V1 不应做成 Windows 效率工具合集。

V1 应定位为：

> 面向重度 Windows 效率用户的 Workspace 工作状态管理工具，用一个入口保存、启动、恢复任务上下文。

核心价值不是更快打开一个应用、文件或网页，而是更快回到一个完整工作状态。

如果 V1 同时做 Launcher、Search、Capture、OCR、Desktop、Clipboard、AI Assistant、Window Manager，产品会失去主线，开发范围也不可控。第一版必须围绕 Workspace 收敛。

## 2. 产品定位

### 2.1 一句话定位

光枢 Lumora 是一个 Windows 工作状态管理系统，帮助用户把分散的应用、文件、网页、命令和窗口布局保存为可恢复的 Workspace。

### 2.2 用户心智

- Everything：找文件。
- Listary / Raycast：启动内容。
- Ditto：管理剪贴板。
- Snipaste / ShareX：截图与贴图。
- PowerToys：系统增强工具箱。
- Lumora：恢复工作状态。

Lumora 不应正面替代这些成熟工具，而应成为它们上层的工作状态控制台。

### 2.3 产品边界

Lumora V1 做：

- 创建 Workspace。
- 启动 Workspace。
- 保存当前工作状态。
- 恢复应用、文件夹、文件、网页、命令。
- 恢复基础窗口布局。
- 提供轻量 Launcher 作为入口。
- 提供轻量截图能力作为辅助。

Lumora V1 不做：

- 完整文件搜索工具。
- 完整截图编辑器。
- 完整剪贴板管理器。
- 桌面清理工具。
- AI 助手主界面。
- 插件市场。
- 云同步。
- 团队管理。
- 企业权限系统。

## 3. 目标用户

### 3.1 核心用户

重度 Windows 效率用户。

他们通常已经使用过或正在使用：

- Everything
- PowerToys
- Listary
- Ditto
- Snipaste / ShareX
- AutoHotkey
- 各类窗口管理工具

他们不缺单点工具，真正的问题是工作状态分散。

### 3.2 用户特征

- 熟悉快捷键。
- 愿意配置工具。
- 同时处理多个任务或项目。
- 经常在浏览器、文件夹、软件、文档、终端之间切换。
- 对效率工具的性能和稳定性敏感。
- 能理解 Workspace、模板、窗口布局等概念。

### 3.3 首批场景用户

#### 程序员 / 技术工作者

典型工作状态：

- IDE
- 终端
- Git 仓库目录
- 本地服务地址
- API 文档
- 浏览器调试页
- 任务管理网页

核心痛点：

- 项目切换成本高。
- 每次进入项目都要重复打开同一批内容。
- 窗口布局和多屏状态容易丢失。

#### 研究 / 写作用户

典型工作状态：

- 文档编辑器
- PDF
- 网页资料
- 笔记软件
- 文件夹
- 截图/摘录内容

核心痛点：

- 资料分散。
- 中断后难以回到之前的阅读和写作上下文。

#### 办公项目用户

典型工作状态：

- 项目文件夹
- Office 文档
- 浏览器后台
- IM 窗口
- 项目管理网页

核心痛点：

- 项目之间来回切换。
- 临时任务打断后恢复成本高。

## 4. 核心问题

目标用户现在的问题不是打不开东西，而是：

> 无法把一个任务所需的上下文作为整体保存和恢复。

具体表现：

- 开工前需要手动打开一批固定内容。
- 任务切换时窗口、文件、网页、命令状态被打散。
- 多个工具各自独立，无法组成稳定工作流。
- 一旦重启电脑或被临时任务打断，需要重新拼装状态。
- 用户知道自己要做什么，但进入状态前有大量机械动作。

Lumora 要解决的是这些机械动作，而不是再做一个更复杂的工具箱。

## 5. 产品主线

V1 的主线只有一条：

> 创建 Workspace → 保存状态 → 启动 Workspace → 恢复状态 → 反馈恢复结果。

所有功能必须服务这条主线。

判断一个功能是否进入 V1，只问一个问题：

> 它是否直接帮助用户更快回到某个工作状态？

如果答案是否定的，不进 V1。

## 6. V1 功能范围

### 6.1 Workspace

Workspace 是 V1 的核心。

#### 功能

- 新建 Workspace。
- 编辑 Workspace 名称、图标、颜色。
- 添加应用。
- 添加文件。
- 添加文件夹。
- 添加网址。
- 添加命令。
- 添加备注。
- 从当前状态生成 Workspace。
- 启动 Workspace。
- 保存上次启动状态。
- 显示启动结果。
- 显示恢复失败项。

#### 启动对象

每个 Workspace 可以包含：

- AppItem：本地应用。
- FileItem：本地文件。
- FolderItem：本地文件夹。
- UrlItem：网页链接。
- CommandItem：命令。
- LayoutItem：窗口位置和大小。

#### 关键规则

- 启动失败必须明确展示。
- 不能静默吞掉错误。
- 不能假装完整恢复。
- 用户必须能看到哪些项成功、哪些失败、为什么失败。

### 6.2 Launcher

Launcher 是 Workspace 的入口，不是产品核心。

#### 功能

- 快捷键呼出。
- 搜索 Workspace。
- 搜索应用。
- 搜索最近启动项。
- 执行命令。
- 快速启动 Workspace。

#### 不做

- 插件市场。
- 复杂命令流。
- 第三方生态集成。
- 浏览器历史深度搜索。
- 内容全文搜索。

Launcher 的目标是够用、快、稳定，不追求替代 Raycast 或 Listary。

### 6.3 Window Layout

Window Layout 只服务 Workspace。

#### 功能

- 保存窗口位置。
- 保存窗口大小。
- 保存所在屏幕。
- 恢复窗口布局。
- 应用未启动时先启动应用，再尝试布局。
- 多屏基础识别。

#### 不做

- 完整窗口管理器。
- 花式分屏。
- 动画吸附。
- 复杂规则系统。
- 独立窗口布局市场。

### 6.4 Capture Lite

截图不是主线，但可以作为效率用户的辅助功能。

V1 保留轻量能力：

- 区域截图。
- 窗口截图。
- 全屏截图。
- 复制到剪贴板。
- 保存到截图历史。
- 简单标注：箭头、矩形、文字、高亮。
- 可选贴图模式。

V1 不做：

- 长截图。
- 复杂图像编辑。
- OCR 工作流。
- AI 分析入口。
- 复杂素材管理。

### 6.5 Settings

V1 设置必须克制。

功能：

- 快捷键设置。
- 开机启动。
- 主题设置。
- Workspace 数据位置。
- 性能模式。
- 启动行为设置。
- 窗口恢复策略设置。

## 7. MVP 流程

### 7.1 手动创建 Workspace

1. 用户按快捷键呼出 Lumora。
2. 输入或选择新建 Workspace。
3. 输入名称，例如“客户 A 项目”。
4. 添加应用、文件夹、文件、网页、命令。
5. 保存。
6. 之后可从 Launcher 直接启动。

### 7.2 从当前状态创建 Workspace

1. 用户已经打开一批相关应用、窗口和网页。
2. 呼出 Lumora。
3. 选择“从当前状态创建 Workspace”。
4. Lumora 读取当前进程、窗口标题、路径、浏览器链接等可识别信息。
5. 用户确认保留项，删除无关项。
6. 保存为 Workspace。

这是 V1 最重要的功能之一，因为它降低了配置成本。

### 7.3 启动 Workspace

1. 用户呼出 Lumora。
2. 搜索 Workspace 名称。
3. 回车启动。
4. Lumora 按顺序打开应用、文件夹、文件、网页和命令。
5. 等待必要窗口出现。
6. 尝试恢复窗口布局。
7. 显示结果。

### 7.4 恢复结果反馈

恢复结束后必须展示：

- 成功启动项。
- 失败启动项。
- 路径不存在项。
- 权限不足项。
- 应用未响应项。
- 窗口布局未恢复项。

这是产品可信度的核心。不能用模糊提示替代。

## 8. 免费版与 Pro

### 8.1 免费版

免费版目标：让用户验证 Workspace 价值。

建议限制：

- 最多 3 个 Workspace。
- 基础 Launcher。
- 基础 Workspace 启动。
- 基础窗口恢复。
- Capture Lite 基础截图。
- 本地数据。

不建议限制性能或核心体验。免费版必须好用，否则用户不会升级。

### 8.2 Pro

Pro 付费点必须围绕 Workspace 深度，而不是堆功能。

Pro 包含：

- 无限 Workspace。
- Workspace 模板。
- 高级窗口布局。
- 多屏布局增强。
- 自动保存工作状态。
- 最近状态恢复。
- 截图历史增强。
- OCR。
- 快捷命令。
- 后续云同步。

### 8.3 企业版

企业版不进入当前设计重点。

未来可能包含：

- 团队 Workspace 模板。
- 统一配置下发。
- IT 管理。
- 权限控制。
- 审计日志。

企业版必须等个人版验证后再设计，否则是空中楼阁。

## 9. 路线图

### V0.1 原型

目标：证明 Workspace 启动链路成立。

范围：

- Workspace 创建。
- 添加应用、文件夹、文件、网址、命令。
- 一键启动 Workspace。
- 快捷键呼出 Launcher。
- 本地配置存储。

验收：

- 用户可以创建 3 个 Workspace。
- 用户可以通过快捷键启动任意 Workspace。
- 启动失败能被看见。

### V0.2 可用版

目标：证明保存当前状态有价值。

范围：

- 从当前状态创建 Workspace。
- 读取当前窗口。
- 基础窗口布局保存。
- 基础窗口布局恢复。
- 恢复结果反馈。
- 基础设置页。

验收：

- 用户可以从当前打开内容生成 Workspace。
- 至少能恢复常见应用窗口位置。
- 失败项明确展示。

### V0.3 增强版

目标：打磨重度用户高频体验。

范围：

- Workspace 模板。
- 最近 Workspace。
- Capture Lite。
- 多屏基础支持。
- 启动顺序设置。
- 延迟启动设置。

验收：

- 用户可以复用模板。
- 多屏用户可以保存和恢复基础布局。
- 截图能力形成基础闭环。

### V1.0 发布版

目标：形成可收费产品。

范围：

- 稳定 Workspace 恢复。
- Launcher 搜索体验打磨。
- Pro 付费边界。
- 自动更新。
- 官网下载。
- 错误日志。
- 用户反馈入口。

验收：

- Workspace 启动和恢复稳定。
- 免费版可验证核心价值。
- Pro 有明确升级理由。

## 10. 成功指标

不要用功能完成数衡量产品。

应使用行为指标：

- 用户每天启动 Workspace 次数。
- Workspace 创建后 7 天仍被使用的比例。
- 每个用户平均 Workspace 数量。
- Workspace 平均恢复成功率。
- 从呼出 Lumora 到启动 Workspace 的平均耗时。
- 启动失败率。
- 用户手动编辑 Workspace 的频率。
- 免费用户触达 Workspace 数量上限的比例。
- Pro 转化率。

核心指标建议：

> 7 天内至少 3 次启动同一个 Workspace 的用户比例。

这个指标能直接验证产品是否真的进入用户工作流。

## 11. 风险清单

### 11.1 范围失控

风险：

V1 继续加入 OCR、AI、Clipboard、Desktop、全文搜索等模块，产品变成浅层工具箱。

处理：

所有功能必须通过 Workspace 价值判断。不直接帮助恢复工作状态的功能不进 V1。

### 11.2 恢复能力不稳定

风险：

Windows 应用差异大，窗口恢复和状态读取不可能 100% 稳定。

处理：

不承诺完整恢复。产品必须如实展示成功项和失败项，用透明性建立信任。

### 11.3 配置成本过高

风险：

用户不愿意手动配置 Workspace。

处理：

优先做“从当前状态创建 Workspace”，让用户从已有工作状态反向生成配置。

### 11.4 与成熟工具正面竞争

风险：

如果尝试替代 Everything、Listary、Snipaste、Ditto，会进入强竞争领域。

处理：

只做上层 Workspace 管理，不追求单点工具极致能力。

### 11.5 AI 误导定位

风险：

AI 功能容易吸引注意力，但不能构成产品核心壁垒。

处理：

AI 后置，只作为 OCR、摘要、文件分析等辅助能力进入 Pro，不进入主界面。

## 12. 产品原则

1. Workspace 是唯一核心。
2. Launcher 是入口，不是主体。
3. Window Layout 是 Workspace 的能力，不是独立窗口管理器。
4. Capture 是辅助能力，不抢主线。
5. 不做工具合集。
6. 不用 fallback 掩盖失败。
7. 恢复失败必须可见。
8. V1 宁愿少，也要形成强记忆点。
9. 免费版必须验证核心价值。
10. Pro 必须围绕 Workspace 深度收费。

## 13. 最终判断

光枢 Lumora 的机会不在于做更多工具，而在于把用户每天重复拼装的工作状态抽象成 Workspace。

第一版必须证明一件事：

> 用户是否愿意每天通过 Lumora 回到自己的工作状态。

如果这个假设成立，再扩展截图、OCR、剪贴板、桌面整理、AI 都有基础。

如果这个假设不成立，其他功能做得再多也只是工具堆叠。
